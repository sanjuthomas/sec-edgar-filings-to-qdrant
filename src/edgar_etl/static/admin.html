Manage Qdrant vector storage, semantic and BM25 search, and Kafka consumption. ETL starts in standby until you start the consumer.

Qdrant — dense vectors & embedding model

Text chunks are converted into dense vectors by the active embedding backend, then stored in the dense named vector on each Qdrant point. Semantic search embeds your question with the same backend and uses cosine similarity.

Qdrant — BM25 keyword search

Qdrant v1.15+ runs server-side BM25 inference. Chunk text is stored in the content-bm25 sparse vector slot on each point — no separate ETL step. Keyword search queries the sparse vector with the Qdrant/bm25 model.
